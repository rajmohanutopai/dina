[CRITICAL-01] Persona unlock passphrase validation is a stub (accepts almost any passphrase)
Evidence
- core/internal/adapter/identity/identity.go:692-698
  - Explicit TODO + warning log says passphrase validation is not implemented.
  - Any passphrase is accepted except sentinel string "WRONG_PASSPHRASE".
Impact
- Locked persona protection can be bypassed by any caller with endpoint access.
- Violates the core cryptographic lock/unlock model.
Exploit path
- Authenticated caller sends POST /v1/persona/unlock with arbitrary non-empty passphrase.
Remediation
- Replace stub with real Argon2id-based verification against stored passphrase hash.
- Treat wrong passphrase as 401/403, not internal error.
- Add tests for correct/wrong/empty passphrase behavior.
Agreed: YES — Verified at identity.go:692-698. The code literally logs "STUB: passphrase validation not implemented" and only rejects the sentinel "WRONG_PASSPHRASE".
Suggested solution (revised — reviewer corrections accepted)
- Add `passphraseHash` (PHC-format string) and `salt` ([]byte) fields to the persona backing store (identity.sqlite), populated at persona creation time via `auth.HashPassphrase(passphrase, salt)` (Argon2id, exists at `core/internal/adapter/auth/auth.go:667`).
- In `PersonaManager.Unlock()`, replace the stub block (lines 692-698) with:
    verifier := auth.NewPassphraseVerifier(p.PassphraseHash)
    ok, err := verifier.Verify(passphrase)
    if err != nil { return fmt.Errorf("persona: verify passphrase: %w", err) }
    if !ok {
        return domain.ErrInvalidPassphrase   // new sentinel error
    }
  Note: `auth.ComparePassphraseHash()` does not exist. The correct API is `auth.NewPassphraseVerifier(storedHash).Verify(passphrase)` which re-derives Argon2id with the stored params/salt and does constant-time comparison internally.
- Persist hash and salt in the persona record in identity.sqlite (not just in-memory), so passphrase protection survives restarts.
- Return HTTP 403 (not 500) from handler when `ErrInvalidPassphrase` is the cause.
- Add rate limiting on the unlock endpoint (max 5 attempts per persona per minute) to prevent brute-force.
- Tests: correct passphrase unlocks, wrong passphrase returns 403, empty passphrase returns 400, rate limit kicks in after 5 failures, passphrase hash survives restart.
FIX STATUS: DONE — Stub replaced with real VerifyPassphrase injection. ErrInvalidPassphrase sentinel added. Handler returns 403.

[CRITICAL-02] Persona DEK derivation at unlock uses raw SHA-256(passphrase+persona), not memory-hard KDF
Evidence
- core/internal/handler/persona.go:108-110
  - DEK derived as sha256("dina-dek:" + passphrase + ":" + persona).
Impact
- Offline guessing cost is too low if related material leaks.
- Does not match architecture intent (Argon2id + HKDF derivation from master seed).
Exploit path
- Compromise of encrypted vault material plus weak passphrase can be brute-forced much faster.
Remediation
- Derive KEK with Argon2id (per-user salt), unwrap master seed, derive per-persona DEK via HKDF.
- Remove direct passphrase->DEK construction entirely.
Agreed: YES — Verified at persona.go:108-110. The line `dek := sha256.Sum256([]byte("dina-dek:" + req.Passphrase + ":" + req.Persona))` is a single SHA-256 hash — trivially brute-forceable at billions of attempts/second.
Suggested solution
- The architecture already has the building blocks wired: `crypto.NewArgon2Deriver()` (KEK), `crypto.NewHKDFKeyDeriver()` (DEK), and `crypto.NewAESGCMKeyWrapper()` (key wrap/unwrap).
- Implementation in persona.go HandleUnlockPersona:
    1. Derive KEK: `kek := kekDeriver.Derive(passphrase, persona.Salt, argon2Params)` (Argon2id, ~100ms target)
    2. Unwrap master seed: `masterSeed := keyWrapper.Unwrap(kek, persona.WrappedSeed)` (AES-256-GCM)
    3. Derive per-persona DEK: `dek := hkdfDeriver.Derive(masterSeed, "persona:"+persona.Name, 32)` (HKDF-SHA256)
    4. Open vault: `vaultManager.Open(ctx, persona, dek)`
- At persona creation time, generate a random 16-byte salt, wrap the master seed under the KEK, and store `{salt, wrappedSeed}` alongside the persona record.
- Migration path (reviewer correction accepted): The “no migration path needed” assumption was incorrect. Persistent vault data exists in SQLCipher databases (config.VaultPath, identity.sqlite, per-persona .sqlite files). The Dockerfile builds with CGO+libsqlcipher and the config struct has a VaultPath field. Migration plan:
    1. Add a `dek_version` field to the persona record (default 1 for SHA-256, 2 for Argon2id+HKDF).
    2. On unlock with version=1: derive DEK using the old SHA-256 path, open vault, re-derive DEK using new path, re-encrypt vault keys under new DEK, update version to 2.
    3. On unlock with version=2: use Argon2id+HKDF path directly.
    4. After successful migration, the old SHA-256 derivation code can be removed in a future release.
    5. If migration fails mid-way, leave version=1 (safe — old path still works).
- Delete the SHA-256 derivation line ONLY after all persona records have been migrated to version=2.
- Tests: verify DEK is deterministic for same passphrase, different for different passphrases, wrong passphrase fails at unwrap step, migration from version=1 to version=2 succeeds, migration failure leaves version=1 intact.
FIX STATUS: DONE — SHA-256 DEK derivation replaced with Argon2id (3 iterations, 128MB, 4 threads, 32-byte output). DEKVersion field added to Persona struct.

[CRITICAL-03] Identity seed handling has a catastrophic fallback to predictable zero/partial seed
Evidence
- core/cmd/dina-core/main.go:79-93
  - bootstrapSeed initialized to zero bytes.
  - DINA_IDENTITY_SEED decoded and copied without length validation (line 87).
  - If identity_seed.hex exists but is invalid/short, code does not fail and does not regenerate, leaving zero seed in memory.
- core/cmd/dina-core/main.go:80-104
  - Seed also persisted in plaintext hex file.
Impact
- Deterministic/predictable identity key generation if seed input is malformed/tampered.
- Full identity compromise potential (DID/signing keys).
Exploit path
- Corrupt/truncate identity seed file or inject short seed env value; node boots with weak predictable seed.
Remediation
- Require exact seed length (32 bytes) and hard-fail on invalid decode/length.
- Eliminate plaintext seed-at-rest; use wrapped seed only.
- Add startup integrity checks and refusal to continue on malformed seed sources.
Agreed: YES — Verified at main.go:79-93. The `copy(bootstrapSeed, decoded)` at line 87 silently pads short seeds with zero bytes from the initial `make([]byte, 32)`. The file-read path (line 90) only requires `len(decoded) >= 32` but short/invalid hex silently leaves the zero seed in place (no error, no exit).
Suggested solution
- After hex decode from env var, add strict length check:
    if len(decoded) != 32 {
        slog.Error("DINA_IDENTITY_SEED must be exactly 32 bytes (64 hex chars)", "got", len(decoded))
        os.Exit(1)
    }
- After hex decode from file, replace the silent fallthrough (line 90-93) with a hard failure:
    if err != nil || len(decoded) != 32 {
        slog.Error("Corrupt identity seed file — refusing to start", "path", seedPath, "error", err)
        os.Exit(1)
    }
- Add an all-zeros check after all seed sources:
    allZero := true
    for _, b := range bootstrapSeed { if b != 0 { allZero = false; break } }
    if allZero {
        slog.Error("Identity seed is all zeros — catastrophic failure, refusing to start")
        os.Exit(1)
    }
- For seed-at-rest protection (revised — reviewer corrections accepted):
    - Do NOT wrap with a machine-specific secret — this breaks container portability, host migration, and backup recovery.
    - Preferred options, in order:
      1. User passphrase (Argon2id-derived KEK wraps the seed) — portable, recoverable.
      2. Managed secret store / KMS (e.g., Docker secrets, Vault, cloud KMS) — enterprise path.
      3. OS keychain (macOS Keychain, Linux Secret Service) — desktop deployments.
    - At minimum: enforce file permissions 0600 on seed file, 0700 on parent directory.
- Do NOT log seed fingerprints in production logs — this is a stable correlatable identifier that serves no operational purpose in normal operation. Seed verification should be a dedicated CLI subcommand (`dina-core --verify-seed`) that runs interactively, not a log line.
FIX STATUS: DONE — Strict 32-byte seed length validation, all-zeros check, hard os.Exit(1) on failure. MaxHeaderBytes added to http.Server.

[CRITICAL-04] D2D transport accepts unsigned legacy inbound payloads; signature verification is optional
Evidence
- core/internal/service/transport.go:321-346
  - Legacy raw sealed box path is accepted when JSON wrapper absent.
- core/internal/service/transport.go:365-401
  - Signature verification only occurs when signature is present.
- core/internal/ingress/sweeper.go:113-137 and 188-214
  - Dead-drop sweeper decrypts/unmarshals and forwards message without signature verification.
Impact
- Sender authenticity can be bypassed for legacy/unsigned payloads.
- Message spoofing risk once ciphertext decrypts successfully.
Exploit path
- Submit raw encrypted payload (no signature) through /msg; message can be accepted without Ed25519 provenance check.
Remediation
- Enforce signature-required inbound format.
- Reject unsigned legacy payloads in production (feature flag only for migration).
- Add signature verification in sweeper before delivery callback.
Agreed: YES — Verified at transport.go:348-359 (legacy fallback) and 379 (signature only verified "if sigHex != ''"). The sweeper at sweeper.go:113-137 decrypts and delivers without any signature check at all.
Suggested solution
- In `TransportService.ProcessInbound()`, after the JSON parse attempt (line 348), add:
    if payload.Sig == "" {
        if os.Getenv("DINA_ALLOW_UNSIGNED_D2D") != "1" {
            return nil, fmt.Errorf("transport: %w: unsigned inbound messages rejected", domain.ErrInvalidSignature)
        }
        slog.Warn("transport: accepting unsigned legacy D2D (migration mode)")
    }
  This makes unsigned rejection the default with an explicit opt-out for migration.
- In the sweeper (`sweeper.go`) (revised — reviewer corrections accepted):
    - Do NOT add a `Signature` field to `domain.DinaMessage`. The signature is envelope-level (`d2dPayload.s`), not message-level. The plaintext DinaMessage is the inner content; the signature covers the transport wrapper.
    - Instead of duplicating decrypt/unmarshal/verify logic in the sweeper, inject a `TransportService` reference and delegate to `ProcessInbound(ctx, blob)`:
        // Replace steps 1-3 (convert keys, decrypt, unmarshal) with:
        msg, err := s.transport.ProcessInbound(ctx, blob)
        if err != nil {
            result.Failed++; continue
        }
      This keeps decryption + signature verification single-sourced in TransportService.
    - The sweeper retains its own TTL check and delivery callback (steps 4-5) since those are sweeper-specific concerns.
    - Dead drop blobs store the raw envelope bytes (JSON wrapper or legacy raw), so ProcessInbound handles both formats correctly.
- Tests: unsigned payload in dead drop → rejected by ProcessInbound, signed payload → delivered, migration flag → unsigned accepted with warning log, sweeper rejects expired messages after ProcessInbound succeeds.
FIX STATUS: DONE — Unsigned rejection default in both JSON wrapper and legacy paths. DINA_ALLOW_UNSIGNED_D2D=1 migration override. Sweeper delegates to TransportProcessor.

[HIGH-05] Public /msg endpoint reads entire body into memory before size checks (DoS)
Evidence
- core/internal/handler/message.go:79
  - io.ReadAll(r.Body) on unauthenticated public endpoint.
- Payload cap is checked only later through ingress router.
Impact
- Memory exhaustion risk from oversized request bodies.
Exploit path
- Repeated oversized POST /msg requests can cause high memory pressure/instability.
Remediation
- Use http.MaxBytesReader at handler boundary before reading.
- Reject oversized bodies with 413 early.
Agreed: YES — Verified at message.go:79. The `io.ReadAll(r.Body)` has no size limit and the `/msg` endpoint is unauthenticated (public D2D ingress). The ingress router's `CheckPayloadSize()` runs after the full body is already in memory.
Suggested solution
- At the top of `HandleIngestNaCl`, replace `io.ReadAll(r.Body)` with a bounded read:
    const maxEnvelopeSize = 256 * 1024  // 256 KB — matches ingress payload cap
    r.Body = http.MaxBytesReader(w, r.Body, maxEnvelopeSize)
    body, err := io.ReadAll(r.Body)
    if err != nil {
        var maxBytesErr *http.MaxBytesError
        if errors.As(err, &maxBytesErr) {
            http.Error(w, `{"error":"payload too large"}`, http.StatusRequestEntityTooLarge)
            return
        }
        http.Error(w, `{"error":"failed to read body"}`, http.StatusBadRequest)
        return
    }
  Note (reviewer correction accepted): Use `errors.As(err, &http.MaxBytesError{})` instead of string comparison on `err.Error()`. The `http.MaxBytesError` type was added in Go 1.19 and the project uses Go 1.25. String comparison is fragile across Go versions.
- This ensures the kernel/Go runtime stops reading at 256KB regardless of Content-Length header.
- Also add a global server-level `ReadTimeout` and `MaxHeaderBytes` on the `http.Server` in main.go if not already set.
- Tests: 1KB payload → 202, 257KB payload → 413, connection closed mid-read → 400.
FIX STATUS: DONE — http.MaxBytesReader(256KB) added to HandleIngestNaCl with errors.As → 413.

[HIGH-06] Signature-auth middleware reads full body before auth with no size bound (DoS)
Evidence
- core/internal/middleware/auth.go:56-59
  - If X-DID/X-Signature/X-Timestamp are present, middleware reads full body using io.ReadAll.
Impact
- Unauthenticated attacker can set those headers and force large body allocation pre-auth.
Exploit path
- Send huge body + fake signature headers -> memory pressure before rejection.
Remediation
- Apply request body size limits before read in auth middleware.
- Fail fast on over-limit bodies.
Agreed: YES — Verified at auth.go:56-59. Any request with `X-DID`, `X-Signature`, and `X-Timestamp` headers triggers `io.ReadAll(r.Body)` before any authentication is performed. An attacker can set these three headers on any endpoint to force unbounded memory allocation.
Suggested solution (revised — reviewer corrections accepted)
- First, validate X-Timestamp BEFORE reading the body (cheap string parse, rejects stale/replayed requests with zero memory allocation):
    if !isTimestampValid(xTS) {
        http.Error(w, `{"error":"invalid or expired timestamp"}`, http.StatusUnauthorized)
        return
    }
- Then, apply a body size limit before the read in the signature verification path. Pass the real `w` (not nil) so the connection is properly closed on oversized bodies:
    const maxSignedBodySize = 1 * 1024 * 1024  // 1 MB
    r.Body = http.MaxBytesReader(w, r.Body, maxSignedBodySize)
    bodyBytes, err := io.ReadAll(r.Body)
    if err != nil {
        var maxBytesErr *http.MaxBytesError
        if errors.As(err, &maxBytesErr) {
            http.Error(w, `{"error":"request body too large"}`, http.StatusRequestEntityTooLarge)
            return
        }
        http.Error(w, `{"error":"failed to read body"}`, http.StatusBadRequest)
        return
    }
  Note: `http.MaxBytesReader(nil, ...)` is incorrect — passing nil means the connection won't be properly closed when the limit is exceeded. Always pass the real `w` ResponseWriter.
- After reading for signature verification, re-arm `r.Body` for downstream handlers:
    r.Body = io.NopCloser(bytes.NewReader(bodyBytes))
- Middleware ordering: timestamp check → body size limit → body read → signature verify → re-arm body → next handler.
- Tests: huge body + valid-looking signature headers → 413, expired timestamp + huge body → 401 (no body read), normal request → passes through.
FIX STATUS: DONE — Timestamp pre-check before body read, http.MaxBytesReader(1MB) with errors.As → 413, isTimestampValid helper added.

[HIGH-09] Core admin reverse proxy forcibly injects BRAIN_TOKEN to backend admin UI
Evidence
- core/internal/handler/admin.go:31-34
- core/cmd/dina-core/main.go:328
  - AdminHandler configured with cfg.BrainToken.
Impact
- Browser/admin flow is translated to brain identity instead of client/admin identity.
- Confuses trust boundaries and can silently re-couple split-brain controls.
Remediation
- Inject CLIENT_TOKEN (or session-derived admin credential), never BRAIN_TOKEN, on /admin proxy hops.
Agreed: YES — Verified at admin.go:31-34: the reverse proxy Director sets `Authorization: Bearer {h.Token}` where Token is cfg.BrainToken (wired at main.go:328 area). This means browser admin requests arrive at the brain with BRAIN_TOKEN identity.

- Change AdminHandler to use cfg.ClientToken instead of cfg.BrainToken:
    adminH := handler.NewAdminHandler(cfg.BrainURL, cfg.ClientToken)

    if v := os.Getenv("DINA_CLIENT_TOKEN"); v != "" { cfg.ClientToken = v }

    Director: func(req *http.Request) {

Reviewer note (partial disagreement)
- Replacing BRAIN_TOKEN with a static CLIENT_TOKEN is still over-privileged and coarse. For a security-first model, prefer per-session identity propagation (cookie/session) or short-lived signed internal tokens instead of static long-lived bearer injection.
Assessment: Reviewer is partially correct. Replacing BRAIN_TOKEN with CLIENT_TOKEN is a meaningful improvement (separates agent and admin trust domains), but the reviewer is right that it's still a static long-lived credential. The per-session cookie approach (already described as the alternative in lines 287-293 above) is the preferred path for production. Recommended staged approach:
  Phase 1 (immediate): Switch from BRAIN_TOKEN to CLIENT_TOKEN — breaks the role collapse.
  Phase 2 (next release): Forward the browser session cookie to brain; brain admin validates session directly. No static token injection at all.
FIX STATUS: DONE — AdminHandler wiring changed from cfg.BrainToken to cfg.ClientToken in main.go.

[HIGH-11] Device revocation does not revoke bearer client tokens in validator
Evidence
- core/internal/service/device.go:127-150
  - RevokeDevice revokes DID key path only when present.
- core/internal/adapter/pairing/pairing.go:303-313
  - PairingManager marks device revoked internally.
- core/internal/adapter/auth/auth.go:116-124, 154-162
  - Token validator has register/lookup but no revoke for client token map.
Impact
- Revoked device may still authenticate with previously issued bearer token (depending on active validator state).
Remediation
- Add explicit client-token revoke path in token validator.
- Wire DeviceService.RevokeDevice to revoke both signature key and bearer token.
Agreed: YES — Verified at auth.go:116-124: `RegisterClientToken` exists but there is no `RevokeClientToken` method. The device.go:136-143 only calls `s.keyRegistrar.RevokeDeviceKey(d.DID)` for the Ed25519 key — the bearer token in the `clientTokens` map remains valid.
Suggested solution
- Add a `RevokeClientToken` method to the tokenValidator:
    func (v *tokenValidator) RevokeClientToken(token string) {
        v.mu.Lock()
        defer v.mu.Unlock()
        hash := sha256Hex(token)
        delete(v.clientTokens, hash)
    }
- Add a `RevokeClientTokenByDevice` method that finds and removes the token by device ID:
    func (v *tokenValidator) RevokeClientTokenByDevice(deviceID string) {
        v.mu.Lock()
        defer v.mu.Unlock()
        for hash, devID := range v.clientTokens {
            if devID == deviceID {
                delete(v.clientTokens, hash)
                return
            }
        }
    }
- Wire it into DeviceService.RevokeDevice after the key revocation:
    if s.tokenRevoker != nil {
        s.tokenRevoker.RevokeClientTokenByDevice(tokenID)
    }
- Add the `TokenRevoker` interface to the port package and inject it into DeviceService.
- Tests: pair device → authenticate with bearer token → revoke device → bearer token rejected (401).
Reviewer note (partial disagreement)
- The map type is `clientTokens[sha256hex(token)] = deviceID` (auth.go), but current call-site wiring passes `deviceName` to `RegisterClientToken` in `DeviceService.CompletePairing` (`RegisterClientToken(resp.ClientToken, deviceName)`), so effective runtime value is device name today.
- `RevokeDevice` receives `tokenID`; therefore revocation cannot directly call `RevokeClientTokenByDevice(tokenID)` without an identity lookup step.
Assessment (reconciled)
- Both points are directionally valid: map-by-identity revocation works, but identity alignment must be explicit.
- Corrected solution:
  - Keep `RevokeClientTokenByDevice(identity string)` (iterates map values, deletes matching entries).
  - In `DeviceService.RevokeDevice`, resolve `tokenID -> PairedDevice` first, then call revocation with the same identity used at registration (currently `PairedDevice.Name`).
  - Prefer long-term cleanup: register with a stable device identifier (not display name), and include/expose that identifier in pairing/revocation flows so auth and pairing use the same key consistently.
FIX STATUS: DONE — TokenRevoker interface added. RevokeClientTokenByDevice iterates clientTokens map. Wired into DeviceService.RevokeDevice via device Name.

[HIGH-12] Export/Import portability path is simulated, not cryptographically real
Evidence
- core/internal/adapter/portability/portability.go:58-76
  - Writes simulated archive content.
- core/internal/adapter/portability/portability.go:148-151

Impact
