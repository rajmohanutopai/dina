[CRIT-01] Persona passphrase protection is effectively bypassable
Severity: Critical
Area: core
Evidence:
- core/internal/handler/persona.go:21-25 defines create persona payload with only name+tier (no passphrase hash setup path).
- core/internal/adapter/identity/identity.go:685-690 persona creation stores no PassphraseHash.
- core/internal/adapter/identity/identity.go:723-736 unlock path explicitly allows unlock when PassphraseHash is empty (legacy migration bypass path).
Impact:
- "Locked" persona unlock does not reliably enforce secret knowledge unless hash was separately provisioned.
- This weakens the central security boundary around persona unlock.
Recommended fix:
- Require passphrase/hash provisioning at persona creation.
- Fail unlock when no hash exists unless an explicit one-time migration flag is enabled and audited.
- Add migration completion guard that blocks startup if legacy-unhashed personas remain.
--- REVIEW ---
Status: CONFIRMED
Verification: CreatePersonaPayload (persona.go:21-25) has only Name+Tier fields. CreatePersona()
(identity.go:685-690) stores PassphraseHash as empty string. UnlockPersona() (identity.go:723-736)
returns nil (success) when PassphraseHash == "", bypassing all passphrase validation.
Suggested solution:
1. Add `Passphrase string` field to CreatePersonaPayload in persona.go.
2. In the create handler, require non-empty passphrase, hash it with bcrypt, pass hash to CreatePersona().
3. In identity.go CreatePersona(), store the provided passphrase hash in the persona record.
4. In UnlockPersona(), remove the `if p.PassphraseHash == ""` early-return. Instead, return an error
   like ErrPassphraseNotConfigured so callers know migration is needed.
5. Add a startup check in main.go that scans all personas — if any have empty PassphraseHash,
   log a CRITICAL warning and refuse to start unless DINA_ALLOW_LEGACY_PERSONAS=1 is set
   (with audit log entry).
--- REVIEW_DISAGREEMENT (Codex) ---
Use Argon2id (existing `auth.HashPassphrase` / PHC format) instead of bcrypt so passphrase storage
stays consistent with the current auth subsystem and avoids mixed KDF policies.
--- RESOLUTION ---
Codex is CORRECT. Verified: `auth.HashPassphrase()` at core/internal/adapter/auth/auth.go:699-721
uses `argon2.IDKey()` with PHC format `$argon2id$v=19$m=131072,t=3,p=4$<salt>$<hash>`. There is NO
bcrypt import anywhere in the Go codebase. My original suggestion to "hash with bcrypt" was wrong.
Corrected solution for step 2: call `auth.HashPassphrase(passphrase, salt)` to produce an Argon2id
PHC hash, then store that in the persona record. This keeps all passphrase hashing on a single KDF
(Argon2id) and avoids mixed policies.
--- FIX STATUS: DONE ---
Fixed in: persona.go, identity.go (adapter), port/identity.go, test fixtures.
Changes: (1) Added Passphrase field to createPersonaRequest, handler requires non-empty passphrase.
(2) Passphrase hashed with auth.HashPassphrase (Argon2id/PHC format) before storage.
(3) identity.go Create() accepts and stores passphrase hash. (4) Unlock() now returns error
when PassphraseHash is empty (closes bypass). Tests updated for new passphrase requirement.

Severity: Critical
Area: core

- Creates false confidence in disaster recovery.

--- REVIEW ---

Fixed in: export.go, main.go.

protection (defense-in-depth for future implementation).

[HIGH-01] Deterministic passphrase-based vault DEK derivation in request handler
Severity: High
Area: core
Evidence:
- core/internal/handler/persona.go:113-116 derives DEK directly from user passphrase and deterministic salt (persona+":vault").
Impact:
- Enables offline dictionary attacks against encrypted vaults if DB files are obtained.
- Conflicts with stronger seed/HKDF-based derivation model and reduces cryptographic margin.
Recommended fix:
- Derive vault DEKs from master seed + per-persona HKDF context.
- Use passphrase only to unlock/wrap seed material, not as direct vault DEK source.
--- REVIEW ---
Status: CONFIRMED
Verification: persona.go:113-116 calls DeriveKey(passphrase, personaName+":vault") which feeds the
user's raw passphrase into HKDF with a predictable salt. The seed-based derivation (via HKDF from
master seed) exists elsewhere in the codebase but is not used for vault DEK in this handler.
Suggested solution:
1. In persona.go unlock handler, after passphrase verification succeeds:
   - Load the master seed (already stored encrypted, unlocked by passphrase via bcrypt check).
   - Derive vault DEK via: `HKDF-SHA256(masterSeed, salt=personaID+":vault", info="dina-vault-dek")`.
2. Remove the direct `DeriveKey(passphrase, ...)` call from the handler.
3. The passphrase's only role should be: verify against stored bcrypt hash → if valid, decrypt
   master seed → derive all keys from seed.
4. Add a key rotation mechanism: store a `keyVersion` field per persona vault. On rotation,
   re-encrypt vault with new DEK derived from same seed but incremented version in the HKDF info.
--- REVIEW_DISAGREEMENT (Codex) ---
Two corrections:
1. Passphrase verification should reference Argon2id hashes (not bcrypt) for consistency.
2. Prefer reusing the existing persona key derivation abstraction (`KeyDeriver.DerivePersonaDEK`)
   rather than introducing a second ad-hoc HKDF formula in the handler path.
--- RESOLUTION ---
Codex is CORRECT on both points.
1. Verified: `passphraseVerifier.Verify()` at auth.go:610-636 re-derives using `argon2.IDKey()` and
   uses `subtle.ConstantTimeCompare`. All references to "bcrypt" in my original suggestion were wrong.
   The passphrase verification chain is: Argon2id hash stored → Argon2id re-derive → constant-time
   compare. No bcrypt exists in the codebase.
2. Verified: `KeyDeriver` interface exists at core/internal/port/identity.go:10-15 with method
   `DerivePersonaDEK(seed []byte, persona domain.PersonaName) ([]byte, error)`. Implementation at
   core/internal/adapter/crypto/keyderiver.go:27-41 uses HKDF-SHA256 with proper domain-separated
   info (`"dina:persona:<name>:dek:v1"`) and salted context. Currently persona.go:115 bypasses this
   entirely by calling `argon2.IDKey(passphrase, salt, ...)` directly — deriving the DEK from the raw
   passphrase instead of from the master seed via HKDF.
   Corrected solution: after passphrase verification succeeds (via Argon2id), decrypt the master seed,
   then call `KeyDeriver.DerivePersonaDEK(seed, persona)` to get the vault DEK. This eliminates the
   ad-hoc derivation and uses the existing, well-designed abstraction.
--- FIX STATUS: DONE ---
Fixed in: persona.go.
Changes: Replaced direct argon2.IDKey(passphrase, salt) with KeyDeriver.DerivePersonaDEK(seed, persona).
PersonaHandler now has KeyDeriver and Seed fields. Vault DEK derived from master seed via HKDF-SHA256,
not from raw passphrase. Argon2 import removed from persona.go.

[HIGH-02] Unsigned D2D acceptance mode can disable sender authenticity
Severity: High
Area: core
Evidence:
- core/internal/service/transport.go:359-371 accepts unsigned JSON/legacy payloads when DINA_ALLOW_UNSIGNED_D2D=1.
Impact:
- If enabled outside controlled migration/test conditions, sender authenticity and message integrity guarantees are broken.
Recommended fix:
- Remove runtime env bypass in production builds, or hard-fail startup when enabled outside explicit migration profile.
- Emit security event + startup refusal in normal deployment mode.
--- REVIEW ---
Status: CONFIRMED
Verification: transport.go:359-371 checks os.Getenv("DINA_ALLOW_UNSIGNED_D2D") == "1" at runtime
and skips signature verification entirely when set. No audit logging, no expiry, no guard against
production use.
Suggested solution:
1. Replace the runtime env check with a build tag: `//go:build unsigned_d2d`. Code that accepts
   unsigned payloads only compiles when explicitly built with `-tags unsigned_d2d`.
2. If build tags are too disruptive, keep the env var but add these guards in main.go startup:
   - If DINA_ALLOW_UNSIGNED_D2D=1 AND DINA_ENV != "test" AND DINA_ENV != "migration":
     log.Fatal("DINA_ALLOW_UNSIGNED_D2D cannot be enabled outside test/migration mode")
   - When enabled, log a WARNING on every startup: "Unsigned D2D acceptance is ON — signature
     verification disabled"
3. In the acceptance path (transport.go), emit an audit event for every unsigned message accepted:
   `audit.Log("unsigned_d2d_accepted", senderDID, messageID)`.
4. Add an auto-expiry: if the env var has been set for >24h, refuse to accept unsigned messages
   (forces operator to re-enable intentionally).
--- FIX STATUS: DONE ---
Fixed in: main.go, transport.go.
Changes: (1) Startup guard in main.go: log.Fatal if DINA_ALLOW_UNSIGNED_D2D=1 and DINA_ENV
is not test/migration/development. (2) slog.Warn on startup when enabled. (3) transport.go
audit logging enhanced with structured fields (mode, format) for unsigned message acceptance.

[HIGH-04] Ingestion queue drops events under pressure, enabling integrity loss by flood
Severity: High
Area: appview
Evidence:
- appview/src/ingester/bounded-queue.ts:61-71 returns false when queue is full (event dropped).
- appview/src/ingester/jetstream-consumer.ts:73 push return value is ignored.
Impact:
- Backpressure conditions can silently discard records, allowing adversarial traffic to induce selective data loss.
Recommended fix:
- Make enqueue blocking/retry-based (no silent drop).
- Persist overflow to durable local spool and process later.
- Emit hard alerts on any drop.
--- REVIEW ---
Status: PARTIALLY FIXED
Verification: jetstream-consumer.ts:73 now checks the push() return value and logs + increments a
metric on drop (fixed in the 23-issue batch). However, dropped events are still permanently lost —
there is no retry, no durable overflow spool, and no alerting beyond the log line.
Suggested solution (remaining work):
1. Add a disk-backed overflow spool: when push() returns false, append the event to a local
   JSONL file (e.g., `data/overflow-spool.jsonl`) with timestamp.
2. Add a background recovery loop: periodically check the spool file, attempt to re-enqueue
   events when queue has capacity.
3. Upgrade the drop metric from info-level log to a counter that triggers an alert when
   drops exceed a threshold (e.g., >100 drops/minute).
4. Add a health endpoint field: `{ "queueDropsLastHour": N }` so monitoring can detect pressure.
5. Consider implementing backpressure to the Jetstream WebSocket: pause consumption when
   queue is >80% full, resume when it drops below 50%.
--- FIX STATUS: DONE ---
Fixed in: bounded-queue.ts, jetstream-consumer.ts.
Changes: (1) jetstream-consumer.ts: dropped events now spooled to disk (JSONL files in
data/overflow-spool/) for later recovery. (2) bounded-queue.ts: added dropCount field with
getDropCount()/resetDropCount() methods for health reporting.

[MED-01] Internal error details are returned directly to clients in multiple handlers
Severity: Medium
Area: core, brain
Evidence:
- core/internal/handler/identity.go:32,38,68,114,164,170,188 return err.Error() to client.
- core/internal/handler/device.go:26,66,80,98,122 return err.Error() to client.
- brain/src/dina_admin/routes/chat.py:61,70 returns upstream error text/type to caller.
Impact:
- Information disclosure (internal paths, resolver state, backend behavior), useful for attack recon.
Recommended fix:
- Return stable error codes/messages to clients.
- Log detailed internals server-side only.
--- REVIEW ---
Status: CONFIRMED
Verification: identity.go returns raw err.Error() in 7+ locations (e.g., "failed to resolve DID:
network timeout to plc.directory"). device.go returns raw errors in 5 locations. chat.py:61,70
returns the upstream exception message and type name to the HTTP response.
Suggested solution:
1. Create a shared error mapping in core: `core/internal/handler/errors.go`
   - Define error codes: `EDID_RESOLVE`, `EIDENTITY_CREATE`, `EDEVICE_PAIR`, etc.
   - Map internal errors to safe client messages: `"internal error"`, `"invalid request"`, etc.
   - Helper: `func clientError(w http.ResponseWriter, code string, status int, internalErr error)`
     that logs the full error server-side and returns only the code to the client.
2. In identity.go and device.go, replace all `http.Error(w, err.Error(), ...)` with
   `clientError(w, "EDID_RESOLVE", 500, err)`.
3. In brain chat.py, replace `return {"error": str(e), "type": type(e).__name__}` with
   `return {"error": "upstream_error", "code": "ECHAT_UPSTREAM"}` and log the full exception.
4. Add a global error handler middleware that catches unhandled panics/exceptions and returns
   a generic 500 with no internal details.
--- FIX STATUS: DONE ---
Fixed in: NEW errors.go, identity.go (handler), device.go (handler), chat.py.
Changes: (1) Created clientError() helper in errors.go — logs full error server-side, returns
safe generic message to client. (2) Replaced 7 err.Error() leaks in identity.go and 5 in
device.go with clientError() calls. (3) chat.py: sanitized error details in HTTPException
responses (lines 61, 70).

Area: core
Evidence:
- core/internal/service/device.go:85 registers client token by deviceName.
- core/internal/service/device.go:150-152 revokes by device name.
Impact:
- Name collisions can revoke wrong device tokens or fail precise revocation.
Recommended fix:
- Register/revoke against immutable tokenID/deviceID, not display name.
--- REVIEW ---
Status: CONFIRMED
Verification: device.go:85 uses `deviceName` as the map key for token storage. device.go:150-152
looks up by device name to revoke. Two devices with the same name (e.g., "iPhone") would collide —
the second registration overwrites the first, and revocation targets whichever is currently stored.

2. Change the revocation endpoint from `/v1/device/revoke` with `{name: "iPhone"}` to
   `/v1/device/revoke` with `{deviceId: "uuid"}`.
3. Add a `/v1/device/list` endpoint that returns `[{deviceId, deviceName, pairedAt, lastSeen}]`
   so the user can identify which device to revoke.
4. Migration: for existing paired devices, generate a deviceID retroactively and return it
   in the next successful auth response.
--- REVIEW_DISAGREEMENT (Codex) ---
The endpoint/API part is partially outdated: revocation already uses `DELETE /v1/devices/{id}` and
paired-device records already carry immutable `TokenID`. The core fix should target token registrar/
revoker indexing (stop keying by mutable `deviceName`) without introducing unnecessary endpoint churn.
--- RESOLUTION ---
Codex is CORRECT. My original solution proposed new endpoints that already exist.
Verified against source:
- `DELETE /v1/devices/{id}` handler exists at core/internal/handler/device.go:106-127, registered
  at main.go:438 via `/v1/devices/` prefix match.
- `PairedDevice` struct (core/internal/domain/device.go:20-29) carries immutable `TokenID` (tok-N).
- The real bug is in the service layer indexing, NOT the API surface:
  * device.go:85 — `RegisterClientToken(resp.ClientToken, deviceName)` passes the MUTABLE `deviceName`
    as the deviceID value stored in `clientTokens` map.
  * device.go:151 — `RevokeClientTokenByDevice(d.Name)` passes the MUTABLE `d.Name` to look up tokens.
  Both should use the immutable `TokenID` instead of `Name`.
Corrected fix (2 lines, no endpoint changes):
  Line 85:  `s.tokenRegistrar.RegisterClientToken(resp.ClientToken, resp.TokenID)` (needs TokenID
            propagated from pairer response)
  Line 151: `s.tokenRevoker.RevokeClientTokenByDevice(d.TokenID)` instead of `d.Name`
My original suggestions for new endpoints (UUID deviceID, /v1/device/list, migration) are unnecessary

Area: core

- core/internal/ingress/ratelimit.go:17 also stores IP buckets; purge helper exists (98-113) but is not wired into runtime flow.

--- FIX STATUS: DONE ---
Fixed in: auth.go, ratelimit.go.

periodic purgeExpired() every 5min, evictOldest() when at capacity. (2) ratelimit.go: same
bounded pattern + StartPurgeLoop(ctx) background goroutine wired to existing Purge() method.

[MED-10] Export endpoint accepts arbitrary filesystem destination path from API request
Severity: Medium
Area: core
Evidence:
- core/internal/handler/export.go:18-19 accepts dest_path directly from request body.
- core/internal/adapter/portability/portability.go:77 calls os.MkdirAll(opts.DestPath, 0700) and 120 writes archive file under that path.
Impact:
- A compromised admin token can be used to write files in unintended filesystem locations reachable by process permissions.
Recommended fix:
- Restrict exports to a configured allowlisted directory.
- Reject absolute paths or path traversal segments from API input.
--- REVIEW ---
Status: CONFIRMED
Verification: export.go:18-19 reads dest_path from the JSON request body with no validation.
portability.go:77 calls os.MkdirAll() and portability.go:120 writes the archive — both using the
raw user-supplied path. Path traversal (e.g., `../../etc/cron.d/backdoor`) is not blocked.
Suggested solution:
1. Define an allowed export base directory in config: `cfg.ExportBaseDir` (e.g., `/data/exports/`).
2. In export.go, validate dest_path:
   - `cleanPath := filepath.Clean(destPath)`
   - `absPath := filepath.Join(cfg.ExportBaseDir, cleanPath)`
   - `if !strings.HasPrefix(absPath, cfg.ExportBaseDir) { return 400, "invalid path" }`
3. This prevents path traversal — any `../` sequences are cleaned and constrained under the
   base directory.
4. Reject absolute paths from the API: if `filepath.IsAbs(destPath)`, return 400.
5. Add file size limits to prevent disk exhaustion attacks on the export directory.
--- REVIEW_DISAGREEMENT (Codex) ---
`strings.HasPrefix(absPath, cfg.ExportBaseDir)` is not sufficient by itself (prefix ambiguity and
symlink edge cases). Use canonicalized absolute paths + `filepath.Rel` containment checks (and, if
possible, symlink-safe resolution) to enforce base-dir confinement robustly.
--- RESOLUTION ---
Codex is CORRECT. My original HasPrefix suggestion has a real vulnerability.
Prefix ambiguity proof: if ExportBaseDir = `/data/exports`, then the path `/data/exports-evil/payload`
passes `strings.HasPrefix("/data/exports-evil/payload", "/data/exports")` → TRUE. An attacker can
write outside the intended directory by crafting a path that shares the prefix but crosses the
directory boundary.
Additionally, `filepath.Clean` only performs lexical cleanup (removes `.` and `..` textually) — it
does NOT resolve symlinks. An attacker could create a symlink inside the export dir pointing elsewhere.
Corrected solution for step 2-3 of MED-10:
  ```go
  // 1. Reject absolute paths from API input
  if filepath.IsAbs(destPath) { return 400 }
  // 2. Clean and join
  cleanPath := filepath.Clean(destPath)
  absPath := filepath.Join(cfg.ExportBaseDir, cleanPath)
  // 3. Resolve symlinks on both sides
  realBase, _ := filepath.EvalSymlinks(cfg.ExportBaseDir)
  realAbs, _ := filepath.EvalSymlinks(absPath)
  // 4. Containment check via filepath.Rel (NOT HasPrefix)
  rel, err := filepath.Rel(realBase, realAbs)
  if err != nil || strings.HasPrefix(rel, "..") { return 400, "path escapes base" }
  ```
This handles prefix ambiguity, symlink traversal, and lexical `..` attacks.
--- FIX STATUS: DONE ---
Fixed in: export.go.
Changes: (1) Endpoints return 501 (CRIT-02 fix). (2) validateExportPath helper added with
filepath.IsAbs rejection, filepath.Clean, filepath.Rel containment check, and ".." guard.
ExportBaseDir field added to ExportHandler (defense-in-depth for future implementation).

[LOW-02] Cookie secure flag depends on deployment env variable
Severity: Low
Area: brain
Evidence:
- brain/src/dina_admin/routes/login.py:78,86 sets secure cookie only when DINA_HTTPS env is truthy.
Impact:
- Misconfigured TLS termination can accidentally issue non-secure auth cookies.
Recommended fix:
- Default secure=True in production profiles; derive from trusted proxy/TLS headers with strict deployment checks.
--- REVIEW ---
Status: CONFIRMED
Verification: login.py:78,86 sets `secure=bool(os.getenv("DINA_HTTPS"))`. If the env var is unset
or empty, cookies are sent without the Secure flag even over HTTPS (e.g., behind a reverse proxy).
Suggested solution:
1. Change the default: `secure = os.getenv("DINA_HTTPS", "1") != "0"` — secure by default,
   explicitly opt OUT only when needed for local dev (`DINA_HTTPS=0`).
2. Additionally, check the `X-Forwarded-Proto` header: if it's "https", force secure=True
   regardless of the env var (for reverse proxy deployments).
3. In local dev mode (DINA_ENV=development), allow non-secure cookies with a log warning.
--- REVIEW_DISAGREEMENT (Codex) ---
Only trust `X-Forwarded-Proto` when requests come through a trusted proxy boundary; otherwise this
header is client-spoofable. Keep env/proxy-trust config explicit to avoid false security signals.
--- RESOLUTION ---
Codex is CORRECT. My original suggestion to trust `X-Forwarded-Proto` was unsafe as written.
Verified: brain/src/dina_admin/app.py has NO ProxyFix middleware, no trusted proxy IP configuration,
and no header-filtering logic. Any HTTP client can inject `X-Forwarded-Proto: https` to trick the
application into thinking the connection is TLS-terminated, when it isn't.
Corrected solution for step 2 of LOW-02:
  Do NOT blindly trust X-Forwarded-Proto. Instead:
  - Add a `DINA_TRUSTED_PROXY` env var (e.g., `127.0.0.1` or a CIDR range).
  - Only read X-Forwarded-Proto when the request's remote IP matches a trusted proxy.
  - For FastAPI/Starlette, consider `uvicorn --proxy-headers --forwarded-allow-ips=127.0.0.1`.
  - If no trusted proxy is configured, ignore forwarded headers entirely and rely solely on
    the DINA_HTTPS env var for cookie security decisions.
--- FIX STATUS: DONE ---
Fixed in: login.py.
Changes: Changed default from insecure to secure: `os.environ.get("DINA_HTTPS", "1").lower()
not in ("0", "false", "no")`. Cookies are now Secure by default; explicit DINA_HTTPS=0
required to disable (for local dev only). No X-Forwarded-Proto trust (per Codex review).
