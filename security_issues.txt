Security Review Findings (2026-02-26)
Scope: core, brain, appview, cli
Status: open findings only (no code changes in this review)

CRITICAL
================================================================================

[CRIT-01] Persona passphrase protection is effectively bypassable
Severity: Critical
Area: core
Evidence:
- core/internal/handler/persona.go:21-25 defines create persona payload with only name+tier (no passphrase hash setup path).
- core/internal/adapter/identity/identity.go:685-690 persona creation stores no PassphraseHash.
- core/internal/adapter/identity/identity.go:723-736 unlock path explicitly allows unlock when PassphraseHash is empty (legacy migration bypass path).
Impact:
- "Locked" persona unlock does not reliably enforce secret knowledge unless hash was separately provisioned.
- This weakens the central security boundary around persona unlock.
Recommended fix:
- Require passphrase/hash provisioning at persona creation.
- Fail unlock when no hash exists unless an explicit one-time migration flag is enabled and audited.
- Add migration completion guard that blocks startup if legacy-unhashed personas remain.

[CRIT-02] Backup/export-import path is non-production placeholder but exposed as real API
Severity: Critical
Area: core
Evidence:
- core/internal/adapter/portability/portability.go:296-300 exports hardcoded dummy file contents (identity-data/config-data/manifest-data).
- core/internal/adapter/portability/portability.go:331-336 import restore returns synthetic DID/persona result, not real restore.
- core/internal/handler/export.go:28-92 exposes /v1/export and /v1/import as normal endpoints.
Impact:
- Creates false confidence in disaster recovery.
- High risk of irreversible data loss during migration/restore operations.
Recommended fix:
- Disable endpoints until full implementation exists, or gate behind explicit non-production flag.
- Implement real archive collection/restore with checksum verification of actual DB files.

HIGH
================================================================================

[HIGH-01] Deterministic passphrase-based vault DEK derivation in request handler
Severity: High
Area: core
Evidence:
- core/internal/handler/persona.go:113-116 derives DEK directly from user passphrase and deterministic salt (persona+":vault").
Impact:
- Enables offline dictionary attacks against encrypted vaults if DB files are obtained.
- Conflicts with stronger seed/HKDF-based derivation model and reduces cryptographic margin.
Recommended fix:
- Derive vault DEKs from master seed + per-persona HKDF context.
- Use passphrase only to unlock/wrap seed material, not as direct vault DEK source.

[HIGH-02] Unsigned D2D acceptance mode can disable sender authenticity
Severity: High
Area: core
Evidence:
- core/internal/service/transport.go:359-371 accepts unsigned JSON/legacy payloads when DINA_ALLOW_UNSIGNED_D2D=1.
Impact:
- If enabled outside controlled migration/test conditions, sender authenticity and message integrity guarantees are broken.
Recommended fix:
- Remove runtime env bypass in production builds, or hard-fail startup when enabled outside explicit migration profile.
- Emit security event + startup refusal in normal deployment mode.

[HIGH-03] Destructive test endpoint can be enabled in runtime via env toggle
Severity: High
Area: core
Evidence:
- core/cmd/dina-core/main.go:452-456 registers POST /v1/vault/clear when DINA_TEST_MODE=true.
Impact:
- Operational misconfiguration can expose bulk data deletion endpoint in live systems.
Recommended fix:
- Build-time test-only registration (build tags) rather than runtime env toggle.
- Add explicit startup hard-fail if test mode is enabled in non-test profile.

[HIGH-04] Ingestion queue drops events under pressure, enabling integrity loss by flood
Severity: High
Area: appview
Evidence:
- appview/src/ingester/bounded-queue.ts:61-71 returns false when queue is full (event dropped).
- appview/src/ingester/jetstream-consumer.ts:73 push return value is ignored.
Impact:
- Backpressure conditions can silently discard records, allowing adversarial traffic to induce selective data loss.
Recommended fix:
- Make enqueue blocking/retry-based (no silent drop).
- Persist overflow to durable local spool and process later.
- Emit hard alerts on any drop.

MEDIUM
================================================================================

[MED-01] Internal error details are returned directly to clients in multiple handlers
Severity: Medium
Area: core, brain
Evidence:
- core/internal/handler/identity.go:32,38,68,114,164,170,188 return err.Error() to client.
- core/internal/handler/device.go:26,66,80,98,122 return err.Error() to client.
- brain/src/dina_admin/routes/chat.py:61,70 returns upstream error text/type to caller.
Impact:
- Information disclosure (internal paths, resolver state, backend behavior), useful for attack recon.
Recommended fix:
- Return stable error codes/messages to clients.
- Log detailed internals server-side only.

[MED-02] Pairing token comparison is not constant-time
Severity: Medium
Area: core
Evidence:
- core/internal/adapter/pairing/pairing.go:365-372 compares token hash byte-by-byte with early exit.
Impact:
- Timing side-channel risk for token validation path.
Recommended fix:
- Use crypto/subtle.ConstantTimeCompare for hash comparison.

[MED-03] Token revocation uses mutable/non-unique device name mapping
Severity: Medium
Area: core
Evidence:
- core/internal/service/device.go:85 registers client token by deviceName.
- core/internal/service/device.go:150-152 revokes by device name.
Impact:
- Name collisions can revoke wrong device tokens or fail precise revocation.
Recommended fix:
- Register/revoke against immutable tokenID/deviceID, not display name.

[MED-04] Rate-limiter state can grow unbounded with attacker-controlled IP cardinality
Severity: Medium
Area: core
Evidence:
- core/internal/adapter/auth/auth.go:434 stores buckets in map keyed by IP with no periodic purge mechanism.
- core/internal/ingress/ratelimit.go:17 also stores IP buckets; purge helper exists (98-113) but is not wired into runtime flow.
Impact:
- Memory pressure/DoS using many spoofed source IPs.
Recommended fix:
- Add bounded/LRU bucket stores with periodic eviction.
- Normalize/validate source IP and cap map cardinality.

[MED-05] Admin UI state-changing routes lack CSRF token/origin enforcement
Severity: Medium
Area: brain
Evidence:
- brain/src/dina_admin/app.py:80-109 authenticates cookie/bearer but no CSRF token check.
- brain/src/dina_admin/templates/base.html:316-321 fetch helper sends same-origin cookies, but no anti-CSRF header/token lifecycle.
Impact:
- SameSite=strict reduces classic cross-site CSRF, but no explicit CSRF control remains a weakness for same-site/subdomain and defense-in-depth.
Recommended fix:
- Add per-session CSRF token and enforce on mutating routes.
- Validate Origin/Referer on cookie-authenticated unsafe methods.

[MED-06] Admin login has no brute-force throttling or lockout
Severity: Medium
Area: brain
Evidence:
- brain/src/dina_admin/routes/login.py:65-77 processes token validation with no rate-limiting, delay, or attempt accounting.
Impact:
- Increases risk of online guessing attacks against static admin token.
Recommended fix:
- Add per-IP and per-account attempt throttling with exponential backoff.
- Emit security telemetry for repeated failures.

[MED-07] Resolver endpoint can crash on invalid JSON input
Severity: Medium
Area: appview
Evidence:
- appview/src/api/xrpc/resolve.ts:43 uses JSON.parse(subjectJson) without try/catch or schema-safe parse wrapper.
Impact:
- Unhandled parse failures can return 500 and be abused for availability degradation.
Recommended fix:
- Parse with guarded try/catch and validate parsed object against strict schema before use.

[MED-08] Cursor safety logic assumes queue order and can over-advance cursor
Severity: Medium
Area: appview
Evidence:
- appview/src/ingester/bounded-queue.ts:100-103 uses queue[0] as queue minimum timestamp.
Impact:
- If queue order is non-monotonic, safe cursor may move forward past older unprocessed events, causing permanent data gaps.
Recommended fix:
- Track true minimum timestamp across queued entries, or enforce strict ordering invariants.

[MED-09] Admin proxy injects long-lived admin token into every proxied request
Severity: Medium
Area: core
Evidence:
- core/cmd/dina-core/main.go:360 sets admin handler token from cfg.ClientToken.
- core/internal/handler/admin.go:31-34 injects Authorization Bearer token on forwarded admin requests.
Impact:
- Increases blast radius if downstream component/logging is compromised.
Recommended fix:
- Prefer short-lived internal credentials (mTLS or ephemeral signed internal auth) instead of reusing user/admin bearer token.

[MED-10] Export endpoint accepts arbitrary filesystem destination path from API request
Severity: Medium
Area: core
Evidence:
- core/internal/handler/export.go:18-19 accepts dest_path directly from request body.
- core/internal/adapter/portability/portability.go:77 calls os.MkdirAll(opts.DestPath, 0700) and 120 writes archive file under that path.
Impact:
- A compromised admin token can be used to write files in unintended filesystem locations reachable by process permissions.
Recommended fix:
- Restrict exports to a configured allowlisted directory.
- Reject absolute paths or path traversal segments from API input.

LOW
================================================================================

[LOW-01] Session store is in-memory, unbounded, and only lazily expires
Severity: Low
Area: brain
Evidence:
- brain/src/dina_admin/routes/login.py:34 _sessions dict global store.
- brain/src/dina_admin/routes/login.py:43-51 expiry cleanup occurs only on validate access path.
Impact:
- Memory growth risk under repeated successful logins; no central session control across workers.
Recommended fix:
- Add bounded session store + periodic cleanup job.
- Consider persistent/session-backend with explicit max sessions per user/device.

[LOW-02] Cookie secure flag depends on deployment env variable
Severity: Low
Area: brain
Evidence:
- brain/src/dina_admin/routes/login.py:78,86 sets secure cookie only when DINA_HTTPS env is truthy.
Impact:
- Misconfigured TLS termination can accidentally issue non-secure auth cookies.
Recommended fix:
- Default secure=True in production profiles; derive from trusted proxy/TLS headers with strict deployment checks.

[LOW-03] Appview observability is stubbed out
Severity: Low
Area: appview
Evidence:
- appview/src/shared/utils/metrics.ts:14-21 no-op metrics implementation.
Impact:
- Security and reliability incidents may go undetected (drop rates, parse failures, abuse patterns).
Recommended fix:
- Implement real metrics backend and mandatory alert thresholds for ingestion and auth failures.

[LOW-04] Insecure defaults in appview config increase accidental exposure risk
Severity: Low
Area: appview
Evidence:
- appview/src/config/env.ts:4 default DB credentials include weak static value.
- appview/src/config/env.ts:8 default Jetstream URL uses ws:// (non-TLS).
- appview/drizzle.config.ts:8 fallback DB URL includes changeme password pattern.
Impact:
- Unsafe defaults can leak into real deployments and weaken transport/credential security.
Recommended fix:
- Remove insecure defaults for production path; require explicit secure values at startup.

[LOW-05] CLI identity directory permissions are not explicitly hardened
Severity: Low
Area: cli
Evidence:
- cli/src/dina_cli/signing.py:47 creates identity directory without explicit mode/chmod.
- cli/src/dina_cli/signing.py:55 hardens private key file, but directory perms depend on process umask.
Impact:
- Under permissive umask, directory metadata/listing may be too broad.
Recommended fix:
- Enforce identity directory mode 0700 after creation.
