# `@dina/crypto-node`

Crypto-primitive adapter for the Node build target. Covers the
Ed25519 / X25519 / secp256k1 / hashes / HKDF / NaCl sealed-box /
Argon2id / CSPRNG surface Dina needs for identity, D2D, and vault
encryption.

**Status:** scaffolded (task 3.20). Port interfaces defined + skeleton
`NodeCryptoAdapter` class with every method throwing
`Not implemented yet` until tasks 3.21–3.30 fill them in.

## Runtime library choices

| Capability | Library | Why |
|---|---|---|
| **Ed25519 sign/verify/deriveFromSeed** | `@noble/ed25519` + `@noble/hashes` (SLIP-0010) | Same libs `@dina/core` uses — guaranteed byte-identical output between pure-core tests and the Node adapter. Pure JS, zero native deps, audited. |
| **X25519 fromEd25519 / scalarMult** | `@noble/curves/ed25519` | Ships with Ed25519→X25519 conversion. Same provenance as Ed25519. |
| **secp256k1 (did:plc rotation)** | `@noble/curves/secp256k1` | Drop-in; Dina doesn't do any exotic secp256k1 flows, just sign/verify + derive. |
| **SHA-256** | `@noble/hashes/sha256` | Stable, fast-enough; matches everywhere else in the codebase. |
| **BLAKE2b(24)** | `@noble/hashes/blake2b` | Used for NaCl sealed-box nonce derivation — **must match libsodium** (not Go's pre-#9 sha512-truncated path). |
| **HKDF-SHA256** | `@noble/hashes/hkdf` | Standard RFC 5869 impl. |
| **NaCl sealed-box** | `libsodium-wrappers` (native) | Faster than TweetNaCl for the high-volume encrypt/decrypt path. libsodium's constant-time guarantees are binding-grade; `tweetnacl`'s are weaker. |
| **Argon2id** | `argon2` (native) | Only reasonable Node option for password KDF. Bound to native libargon2; prebuilt binaries for linux-x64 / linux-arm64 / darwin-x64 / darwin-arm64. |
| **randomBytes** | `node:crypto.randomBytes` | Node built-in CSPRNG — no npm dep needed. |

### Why pure-JS @noble/* for the bulk of it

We already pulled `@noble/*` into `@dina/core` (protocol-level crypto
helpers + canonical-sign tests). Reusing them here gives us:

1. **Byte-for-byte output parity between pure core and Node adapter.** If
   `@dina/core`'s canonical-sign test emits signature `sig-X`, `@dina/crypto-node.sign()` must produce the same `sig-X` for the same inputs. Sharing the implementation removes a class of cross-runtime bugs.
2. **Zero native-dep surface for CI.** Node can run `@noble/*` anywhere
   without prebuild binaries or toolchain. Reduces install flakiness.
3. **Smaller package install size.** Native bindings add ~MB per
   platform; pure JS stays under 100KB.

### What we gave up by not using native bindings everywhere

- Marginal Ed25519 speed. `@noble/ed25519` is ~2–5× slower than
  `node-sodium` or native `libsodium.crypto_sign_detached`. For Dina's
  sign-per-request workload (Brain→Core HTTP, D2D envelopes), this
  is 50µs vs 10µs — irrelevant at request cadence.
- We DO pay the native-bindings cost for NaCl sealed-box (libsodium)
  and Argon2id (argon2), where the perf delta does matter:
  sealed-box on every inbound D2D message, Argon2id on every persona
  unlock. Hybrid approach — native where it matters, pure-JS elsewhere.

## Cross-runtime conformance (task 3.30)

Dina runs on multiple platforms: Go home node (mobile server-side
via gomobile + dina-core), Node home node (this package's consumer),
Expo mobile app. All three must produce byte-identical output for
the same input. Task 3.30 cross-checks this module's outputs against
hex fixtures generated by Go Core for:

- Ed25519 seed → pubkey + privkey bytes
- Ed25519 sign(privkey, message) → signature bytes
- X25519(Ed25519 → X25519) → expected bytes
- SHA-256(message) → digest
- BLAKE2b(24, message) → 24-byte digest
- HKDF-SHA256(ikm, salt, info, outLen) → derived bytes
- NaCl sealed-box seal/open round-trip
- Argon2id(password, salt, params) → derived key (deterministic given
  fixed params)

Fixtures live in `packages/fixtures/` (Phase 0a landed that package
with initial vectors). A divergence here is **always a correctness
bug in this adapter** — the Go port is the reference.

## Port contract (planned → Phase 2 task 2.1b)

`@dina/core` will declare these ports in Phase 2 task 2.1b. Until
then, the interfaces live here (`Ed25519Port`, `X25519Port`, etc.).
When the canonical declarations land, this package will re-export
under the core-canonical names.
