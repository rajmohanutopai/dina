[APPVIEW-HIGH-01] Web rate limiting is bypassable and memory-unbounded
Evidence:
- appview/src/web/server.ts:15-16, 51-54
Issue:
- The limiter trusts `x-forwarded-for` unconditionally and uses a plain `Map` with no max size.
- Attackers can spoof a new `x-forwarded-for` each request, bypass per-IP limits, and force unbounded key growth within the 60s window.
Impact:
- Effective rate-limit bypass + memory pressure DoS on the public API.
Recommended fix:
- Trust proxy headers only when an explicit trusted proxy mode is enabled.
- Default to `req.socket.remoteAddress`.
- Replace `Map` with bounded LRU (max keys) and per-key TTL.

--- SOLUTION ---
Status: CONFIRMED
Verified: server.ts:51-53 trusts x-forwarded-for unconditionally with remoteAddress fallback.
  Line 15: `rateLimitMap = new Map<>()` has no max size.  Lines 30-37 clean stale entries
  every 60s, but attacker can exhaust memory between cleanup cycles by rotating IPs.

Fix (web/server.ts):
  1. Add trusted proxy guard:
       const TRUST_PROXY = process.env.TRUST_PROXY === '1'
       const clientIp = TRUST_PROXY
         ? (req.headers['x-forwarded-for'] as string)?.split(',')[0]?.trim()
           || req.socket.remoteAddress || 'unknown'
         : req.socket.remoteAddress || 'unknown'

  2. Replace Map with bounded LRU — use lru-cache (already in package.json):
       import { LRUCache } from 'lru-cache'
       const rateLimitMap = new LRUCache<string, { count: number; resetAt: number }>({
         max: 50_000,
         ttl: 60_000,
       })
     This auto-evicts stale entries and caps memory at 50K keys.
--- END SOLUTION ---

[APPVIEW-HIGH-02] Queue failure latches cursor progress (replay storm / ingest stall)
Evidence:
- appview/src/ingester/bounded-queue.ts:35, 94-113, 189
- appview/src/ingester/jetstream-consumer.ts:251-253, 344-346
Issue:
- Failed timestamps are added to `failedTimestamps` but never cleared or retried.
- `getSafeCursor()` includes these failed timestamps forever.
Impact:
- A single persistent handler error can pin cursor advancement, causing repeated replay windows, duplicate work, and long-lived ingest instability.
Recommended fix:
- Add bounded retry policy + dead-letter queue.
- Clear failed timestamps when event is retried successfully or explicitly quarantined.
- Persist failure metadata for operator visibility.

--- SOLUTION ---
Status: CONFIRMED
Verified: bounded-queue.ts:35 `failedTimestamps = new Set<number>()` — never cleared.
  Line 189: failed items add timestamps.  Lines 94-113: getSafeCursor() includes all
  failed timestamps in minimum calculation, permanently blocking cursor advancement.
  No retry, no cleanup, no dead-letter mechanism anywhere in the file.

Fix (ingester/bounded-queue.ts):
  1. Add bounded retry with max attempts:
       private failedAttempts = new Map<number, number>()  // timestamp -> attempt count
       private static MAX_RETRY = 3

  2. In processItem catch block, track attempts:
       const attempts = (this.failedAttempts.get(item.timestampUs) ?? 0) + 1
       if (attempts >= BoundedQueue.MAX_RETRY) {
         this.failedTimestamps.delete(item.timestampUs)
         this.failedAttempts.delete(item.timestampUs)
         logger.error({ timestampUs: item.timestampUs, attempts },
           'Event dead-lettered after max retries')
         // Optionally write to dead-letter spool file for operator review
       } else {
         this.failedAttempts.set(item.timestampUs, attempts)
         this.failedTimestamps.add(item.timestampUs)
       }

  3. Add periodic cleanup for old failed timestamps (e.g., older than 1 hour):
       setInterval(() => {
         const cutoff = Date.now() * 1000 - 3_600_000_000  // 1 hour in microseconds
         for (const ts of this.failedTimestamps) {
           if (ts < cutoff) {
             this.failedTimestamps.delete(ts)
             this.failedAttempts.delete(ts)
           }
         }
       }, 300_000)  // every 5 minutes
--- END SOLUTION ---

[APPVIEW-HIGH-03] Overflow spool is write-only (data-loss under pressure)
Evidence:
- appview/src/ingester/jetstream-consumer.ts:180-195 (write path)
- Search confirms no read/drain path for spool files in appview/src.
Issue:
- Queue overflow events are written to spool files but never replayed.
Impact:
- Under sustained load, dropped events are effectively lost despite "spooling" behavior.
Recommended fix:
- Implement deterministic spool replay (startup + periodic drain) with delete-on-success.
- Track replay offsets and replay metrics.

--- SOLUTION ---
Status: CONFIRMED
Verified: jetstream-consumer.ts:180-195 writes spool files.  Grep of entire appview/src
  for spool read/consume/drain returns zero matches.  Lines 95-111 only clean up stale
  files (>24h) — no consumer.  Events spooled during overload are effectively lost.

Fix (ingester/jetstream-consumer.ts):
  1. Add startup spool replay before connecting to Jetstream:
       private async replaySpool(): Promise<void> {
         const files = readdirSync(SPOOL_DIR).filter(f => f.endsWith('.jsonl')).sort()
         for (const file of files) {
           const path = join(SPOOL_DIR, file)
           const lines = readFileSync(path, 'utf-8').split('\n').filter(Boolean)
           for (const line of lines) {
             try {
               const event = JSON.parse(line)
               await this.handleEvent(event)
             } catch (err) {
               logger.warn({ file, err }, 'Spool replay: skipping malformed event')
             }
           }
           unlinkSync(path)  // delete on successful replay
           logger.info({ file, events: lines.length }, 'Spool file replayed')
         }
       }

  2. Call replaySpool() in start() before WebSocket connect:
       async start() {
         await this.replaySpool()
         this.connect()
       }

  3. Add periodic drain (every 5 min) for files accumulated during runtime.
--- END SOLUTION ---

[APPVIEW-HIGH-07] Future-dated records can poison scoring math
Evidence:
- appview/src/ingester/record-validator.ts:51 (datetime format only, no temporal bounds)
- appview/src/scorer/algorithms/trust-score.ts:81-83, 92
- appview/src/scorer/algorithms/sentiment-aggregation.ts:59-66
Issue:
- `createdAt` can be far in the future; recency uses `exp(-age/halflife)`, so negative age increases weight above 1.
- Extreme values can destabilize weighting (including NaN/Inf propagation risks).
Impact:
- Score manipulation and integrity degradation from timestamp poisoning.
Recommended fix:
- Reject records with `createdAt` outside an allowed skew window.
- Clamp recency weight to a safe range.
- Add NaN/Inf guards before persisting scores.

--- SOLUTION ---
Status: CONFIRMED
Verified: record-validator.ts:51 — isoDateString validates format only, no temporal bounds.
  trust-score.ts:155-157 — daysSince() returns negative for future dates.
  trust-score.ts:81-83 — Math.exp(-ageDays / HALFLIFE) with negative ageDays yields
  values > 1.0, inflating trust weights beyond normal [0,1] range.
  sentiment-aggregation.ts:59-66 — same pattern.

Fix (three layers):
  1. Temporal bounds at ingestion (record-validator.ts):
       const MAX_CLOCK_SKEW_MS = 5 * 60 * 1000  // 5 minutes
       // Add to each schema that uses isoDateString for createdAt:
       createdAt: isoDateString.refine(
         (val) => new Date(val).getTime() <= Date.now() + MAX_CLOCK_SKEW_MS,
         { message: 'createdAt cannot be more than 5 minutes in the future' }
       )

  2. Clamp recency weight in scoring (trust-score.ts and sentiment-aggregation.ts):
       const ageDays = Math.max(0, daysSince(a.recordCreatedAt))  // clamp to non-negative
       const recency = Math.exp(-ageDays / CONSTANTS.SENTIMENT_HALFLIFE_DAYS)

  3. NaN/Inf guard before persisting scores:
       const finalScore = Number.isFinite(score) ? score : 0
--- END SOLUTION ---

[APPVIEW-HIGH-08] Out-of-order replay can overwrite newer state (event revision not enforced)
Evidence:
- appview/src/shared/types/jetstream-types.ts:7, 22 (`commit.rev` exists)
- appview/src/ingester/jetstream-consumer.ts:243-247 (update path ignores `rev`)
- appview/src/ingester/handlers/attestation.ts:66-92 (unconditional upsert update)

Impact:

Recommended fix:
- Persist commit revision/sequence per record and reject stale updates (`incoming_rev <= stored_rev`).
- Add integration tests for out-of-order update replay.

--- SOLUTION ---
Status: CONFIRMED (with nuance — upsert is idempotent for same-version replays,
  but out-of-order replays of different versions can overwrite newer state)
Verified: jetstream-consumer.ts:243-245 routes updates through handleCreate.
