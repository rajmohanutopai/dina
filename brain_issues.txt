- When enabled, `_get_client_ip` trusts the leftmost XFF IP without trusted-hop/CIDR validation (`brain/src/dina_admin/routes/login.py:34-37`).
Why this matters
- Login throttling and audit IP attribution can be bypassed/spoofed if direct access remains possible.
Recommended fix
- Replace boolean with trusted proxy CIDR list.
- Parse XFF from right-to-left and pick first non-trusted hop.
--- SUGGESTED SOLUTION ---
Mirror the pattern already implemented in Go core (`core/internal/middleware/ratelimit.go`).
1. In `brain/src/dina_admin/routes/login.py`, replace boolean with CIDR list:

     _TRUSTED_CIDRS: list[ipaddress.IPv4Network | ipaddress.IPv6Network] = []
     _raw = os.environ.get("DINA_TRUSTED_PROXIES", "")
     if _raw:
         for cidr in _raw.split(","):
             cidr = cidr.strip()
             if cidr:
                 _TRUSTED_CIDRS.append(ipaddress.ip_network(cidr, strict=False))
2. Rewrite `_get_client_ip` to walk XFF right-to-left:
     def _get_client_ip(request: Request) -> str:
         if not _TRUSTED_CIDRS:
             return request.client.host if request.client else "unknown"
         xff = request.headers.get("x-forwarded-for", "")

                 except ValueError:
                     continue
                 if not any(addr in net for net in _TRUSTED_CIDRS):
                     return ip_str
         return request.client.host if request.client else "unknown"

   for consistency with the Go core config.

--- END SOLUTION ---

`_get_client_ip()` now walks XFF right-to-left, returning first non-trusted IP.

[MEDIUM-07] Multiple admin endpoints allow unbounded field lengths; no global body cap
Evidence
- `LoginRequest.token` has no max length (`brain/src/dina_admin/routes/login.py:85-87`).
- `ContactCreate.did/name` have no max length (`brain/src/dina_admin/routes/contacts.py:39-43`).
- `PairCompleteRequest.code/device_name/public_key_multibase` have no max length (`brain/src/dina_admin/routes/devices.py:38-40`).
- Brain container starts uvicorn with no request-body cap flags (`brain/Dockerfile:15`).
Why this matters
- Oversized request bodies/fields can amplify memory and CPU pressure.
Recommended fix
- Add `max_length` constraints for user-controlled strings.
- Enforce body-size limits at app middleware and/or ingress proxy.
--- SUGGESTED SOLUTION ---
Two layers: field-level validation + global body cap.
1. Add `max_length` to Pydantic models:
   - `login.py`: `token: str = Field(..., max_length=512)`
   - `contacts.py`: `did: str = Field(..., max_length=256)`, `name: str = Field(..., max_length=128)`
   - `devices.py`: `code: str = Field(..., max_length=64)`, `device_name: str = Field(..., max_length=128)`,
     `public_key_multibase: str | None = Field(None, max_length=256)`
2. Add body-size limiting middleware to both apps (brain + admin):
     from starlette.middleware import Middleware
     MAX_BODY = 1 * 1024 * 1024  # 1 MiB

     @app.middleware("http")
     async def limit_body_size(request: Request, call_next):
         cl = request.headers.get("content-length")
         if cl and int(cl) > MAX_BODY:
             return JSONResponse(status_code=413, content={"detail": "Request body too large"})
         return await call_next(request)
3. Alternatively, add `--limit-max-request-size` to uvicorn in Dockerfile:
     CMD ["uvicorn", ..., "--limit-max-request-size", "1048576"]
   (uvicorn default is 1 MiB but it's better to make it explicit.)
Files: login.py, contacts.py, devices.py, app.py (×2), Dockerfile (5-6 files, ~20 lines changed)
--- END SOLUTION ---

--- REVIEWER DISAGREEMENT ---
I partially disagree with the implementation details:
- Field-level max lengths are correct.
- `Content-Length`-only checks are bypassable (chunked/no header payloads).
- `--limit-max-request-size` is not a valid uvicorn option in this environment.
Use an enforced streaming/body-read cap at application middleware (actual bytes read) and/or strict ingress proxy limits.
--- END REVIEWER DISAGREEMENT ---

--- ORIGINAL AUTHOR RESPONSE ---
Reviewer is CORRECT on both points.

1. `Content-Length`-only check is bypassable: HTTP/1.1 chunked transfer encoding omits
   the Content-Length header entirely. A malicious client can stream an arbitrarily large
   body through the Content-Length guard. My solution step 2 is insufficient.

2. `--limit-max-request-size` is NOT a valid uvicorn CLI option. Verified: uvicorn's
   documented CLI flags (v0.27+) include `--limit-concurrency` and `--limit-max-requests`
   (for worker restart), but no body-size flag. My solution step 3 is wrong.

Corrected solution:
1. Keep the Pydantic `max_length` field constraints (step 1 of original) — these are correct.
2. Replace the Content-Length-only middleware with a streaming body-read cap:
     MAX_BODY = 1 * 1024 * 1024  # 1 MiB

     @app.middleware("http")
     async def limit_body_size(request: Request, call_next):
         if request.method in ("POST", "PUT", "PATCH"):
             body = await request.body()  # Starlette reads + caches the full body
             if len(body) > MAX_BODY:
                 return JSONResponse(status_code=413, content={"detail": "Request body too large"})
         return await call_next(request)
   `request.body()` reads ALL bytes regardless of Content-Length header, so chunked
   transfer encoding cannot bypass this check.
3. Remove the `--limit-max-request-size` Dockerfile step entirely.
4. For defense-in-depth, enforce body limits at the reverse proxy layer (nginx/Caddy)
   if deployed behind one.
Files: app.py (×2), login.py, contacts.py, devices.py (5 files, ~25 lines changed)
--- END ORIGINAL AUTHOR RESPONSE ---

--- FIX STATUS: DONE ---
Fixed per reviewer recommendation: Added Field(max_length=...) to all Pydantic models
(login.py, contacts.py, devices.py). Added streaming body-size middleware (`request.body()`
cap at 1 MiB) to both brain and admin apps. No Content-Length-only check, no uvicorn flag.
--- GAP CLOSURE (validation round 2) ---
Content-Length header pre-check added as first guard. For the enforced guard,
