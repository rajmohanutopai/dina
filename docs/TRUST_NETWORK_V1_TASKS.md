## Headline finding

**The existing AppView is significantly more sophisticated than the plan assumed.** A first sweep of `appview/src/` shows:

- Lexicon namespace is `com.dina.trust.*` (not `com.dina.subject.*` as the plan says) and covers 19 record types
- `subjects` schema exactly matches the plan's three-tier identifier graph
- `attestations` schema has more structured fields than the plan specs (category, sentiment, domain, dimensions, evidence, mentions, related, bilateralReview, tags, text)
- `subject_scores`, `subject_claims`, `did_profiles`, `domain_scores`, `trust_edges`, `mention_edges` already exist
- Scorer jobs already implement: refresh-subject-scores, refresh-reviewer-stats, decay-scores, detect-sybil, detect-coordination, refresh-domain-scores, refresh-profiles, cleanup-expired, process-tombstones
- Algorithms exist for: trust-score, reviewer-quality, sentiment-aggregation, anomaly-detection, recommendation
- xRPC: get-attestations, get-graph, get-profile, resolve, search, service-search, service-is-discoverable

**Implication:** Phase 0 of the plan is ~70% already implemented. The realistic V1 delta is much smaller. Many tasks below carry status `[?]` and a "verify against plan" sub-task, because the right move is alignment — not reimplementation.

**Recommendation before any code changes**: complete §0 reconciliation tasks first. Several of them may collapse the rest of the doc by 30–40%.
