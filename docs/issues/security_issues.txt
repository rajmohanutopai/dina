- Move to explicit allowlist per scope (not denylist).
- For device scope, deny /v1/persona*, /v1/personas, /v1/pair/*, /v1/export*, /v1/import*.
- Add authz tests for every routed endpoint.
--- SUGGESTED SOLUTION ---

   Device-scope tokens should only be permitted on:
     /v1/msg/send, /v1/msg/inbox, /v1/vault/query, /v1/vault/store,
     /v1/vault/kv, /v1/contacts, /healthz, /readyz

   and asserts the expected 200/403 for each (scope=admin, scope=device, scope=brain).
3. Wire the allowlist via a map[string][]string keyed by scope, loaded at startup from

[SEC-HIGH-02] Pairing race can return mismatched TokenID
Severity: HIGH
Evidence:
- core/internal/adapter/pairing/pairing.go:205-220
  CompletePairingFull() calls CompletePairing(), then re-locks and reads pm.devices[n-1].tokenID.
Impact:
- Concurrent pairings can cause TokenID mismatch between returned ClientToken and returned TokenID.
- Revocation/accountability can target wrong device.
Fix direction:
- Return TokenID atomically from CompletePairing() (same critical section as token generation).
- Do not infer TokenID by reading "last device".
--- SUGGESTED SOLUTION ---
Status: CONFIRMED
1. Change CompletePairing() signature to return (clientToken string, tokenID string, err error).
   Inside the existing critical section (pairing.go:155-200) where the device is appended and
   the token is generated, capture tokenID from the newly created device before releasing the lock.
2. Remove CompletePairingFull() entirely — it exists only to work around the missing return value.
   Callers (handler) use the returned tokenID directly.
3. Add a concurrent pairing test: launch N goroutines calling CompletePairing simultaneously
   and assert every returned (clientToken, tokenID) pair is self-consistent.
Files: core/internal/adapter/pairing/pairing.go, core/internal/handler/device.go (caller).

--- SUGGESTED SOLUTION ---

[SEC-MED-08] AppView ingest path trusts incoming Jetstream events without cryptographic provenance verification
Severity: MEDIUM
Evidence:
- appview/src/ingester/jetstream-consumer.ts:73,119-135,146-171
  Pipeline processes did/commit payload directly after schema validation.
Impact:
- If upstream relay/feed is compromised, forged records can be ingested.
Fix direction:
- Add signature/provenance verification against AT Proto commit/auth chain before persistence.
--- SUGGESTED SOLUTION ---
Status: CONFIRMED
1. Phase 1 (TLS trust): Validate that the Jetstream WebSocket connection uses TLS and
   pin to known Bluesky relay hostnames (JETSTREAM_ALLOWED_HOSTS env var). Reject
   connections to unexpected hosts.
2. Phase 2 (commit verification): For each incoming event, verify the AT Proto commit
   signature using the DID's signing key (resolved via DID document). The @atproto/crypto

   CIDs to detect duplicate or out-of-order events from a compromised relay.
4. Note: Full commit chain verification requires resolving the DID document for each
   event author, which adds latency. Consider async verification with a grace period
   (ingest immediately, verify in background, tombstone on failure).
