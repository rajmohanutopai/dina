- [x] **TN-DB-002** — Done 2026-04-30. New table `appview/src/db/schema/reviewer-namespace-scores.ts`, registered in `db/schema/index.ts`. **Why a SEPARATE table** rather than extending `did_profiles` with a `(did, namespace)` composite PK: most users never use namespaces, so a NULL-bearing `namespace` column would either bloat every row OR force a sentinel like `''` for the root identity (both lossy); the existing `did_profiles.did` PK is referenced by 50+ code paths and changing it is invasive; a dedicated table captures the "namespace-using subset" without polluting root-identity stats. New `appview/src/db/fts_columns.ts` — single-source-of-truth idempotent DDL helper for both FTS surfaces. **Audit verdict on existing state**: `attestations.search_vector` already existed at runtime via inline `db.execute(sql\`ALTER TABLE...\`)` blocks duplicated in BOTH `web/server.ts` AND `ingester/main.ts` (two copies that drift on every change); `subjects` had no FTS column at all; `services` had only a btree-on-search_content for ILIKE. **What landed**: (1) `FTS_DDL_STATEMENTS` — frozen ordered list of 4 idempotent DDL statements covering both `attestations.search_vector` (already existed, now centrally owned) AND new `subjects.search_tsv` plus their GIN indexes; (2) `ensureFtsColumns(db)` — runs every statement against the DB, handles `42P01` (undefined_table) per-statement so a partial cold-boot (web racing ahead of ingester schema-push) still lands the FTS surfaces for tables that DO exist (caller retries on next boot via `IF NOT EXISTS`); (3) both `web/server.ts` and `ingester/main.ts` now call `ensureFtsColumns(db)` instead of duplicating the DDL inline. **Why GENERATED columns are NOT in the Drizzle schema**: declaring `searchVector: tsvector('search_vector')` in `attestations.ts` would make every `db.select().from(attestations)` (no explicit column list — used in 18+ call sites including `api/xrpc/search.ts:287` and every scorer job) pull a fat tsvector into the result row, ballooning response sizes and changing wire shapes. The GIN index requires STORED — VIRTUAL columns can't be indexed. Pinned by test. **Why 'english' dictionary, not per-row language**: V1 cohort is overwhelmingly English; per-row dictionary via CASE in the GENERATED clause is a non-breaking V2 schema change. **Deep review caught + fixed**: initial 42P01 logic bailed at first occurrence, which meant a missing `attestations` table would silently skip the `subjects` FTS DDL (and vice-versa). STORED` on both ALTERs (STORED required for GIN), `to_tsvector('english', coalesce(...))` NULL-safety guard, GIN operator class on both indexes, frozen-at-runtime guard.
