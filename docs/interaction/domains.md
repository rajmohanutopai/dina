# Dina Domains

## Purpose

This document defines the consolidated conceptual domains in which a user may interact with Dina.

The raw idea space is 99 interaction areas (see `docs/interaction/full_interaction_areas.md`). After expansion and consolidation, the interaction surface is organized into 15 primary domains that are small enough to reason about and broad enough to cover the full system.

These domains are intended to support:

- product navigation
- permissions and approvals
- prompt and tool design
- delegated-task routing
- admin and audit design

## The 16 Primary Domains

### 1. Memory and Knowledge

Purpose:

- store, retrieve, and organize durable personal or work context

Includes:

- personal memory
- personal knowledge and Q&A
- document handling
- knowledge ingestion
- learning and preferences
- reflection and journaling

Examples:

- "Remember that I prefer aisle seats."
- "What did I say about ergonomic chairs last month?"

Covers: Personal Memory & Preferences, Knowledge Ingestion, Journaling & Reflection, Photography & Memories, Passwords & Account Recovery, Legacy & Estate Planning

## Cross-Cutting Interaction Modes

These are conceptual interaction modes, not slash commands.

They describe what kind of interaction is happening, regardless of whether the surface is:

- Telegram
- CLI
- admin UI
- OpenClaw
- delegated task execution
- D2D workflow

### User Interaction Modes

These are the primary user-facing intent verbs.

1. `remember`

- store something durable

2. `ask`

- retrieve, explain, summarize, or interpret

3. `plan`

- organize next steps or structure information

4. `act`

- perform an external action or system mutation

5. `monitor`

- watch for change and notify later

The same domain may support multiple user interaction modes.

Examples:

- Memory + remember
- Finance + ask
- Travel + plan
- Communication + act
- Scheduling + monitor

### Operator and Control Modes

These are separate from user interaction modes.

They represent operational and governance actions around the system itself.

1. `inspect`

- check state, status, history, or current conditions

Examples:

- check task status
- inspect queue state
- view session state

2. `approve`

- explicitly authorize or deny something

Examples:

- approve an action
- deny a proposal
- unlock a restricted flow

3. `manage`

- configure or administer the system

Examples:

- pair a device
- configure policy
- revoke access

4. `debug`

- diagnose why something happened or failed

Examples:

- inspect traces
- understand why a task failed
- inspect message flow

5. `recover`

- repair or reconcile broken or stuck state

Examples:

- requeue a task
- reconcile a running task
- end a stale session

### Why The Split Matters

The five user interaction modes are the correct abstraction for intent modeling.

They are not enough to represent all slash command families or operator surfaces.

Commands such as:

- status
- approve
- pair
- retry
- reconcile

are better modeled as operator and control modes, not user intent modes.

## Sensitivity Model

The domain model should work with a separate sensitivity classification:

- `normal`
- `sensitive`
- `restricted`

Recommended default sensitivity by domain:

1. Memory and Knowledge: mixed
2. Tasks and Execution: normal
3. Scheduling and Time: normal
4. Communication: sensitive
5. Shopping, Search, and Recommendations: normal
6. Finance and Assets: sensitive
7. Health and Wellbeing: sensitive
8. Home and Property: mixed
9. Travel and Mobility: mixed
10. Relationships and Social: sensitive
11. Family and Care: sensitive
12. Safety, Permissions, and Governance: restricted
13. Hobbies, Creativity, and Lifestyle: normal
14. Community and Social Impact: normal
15. Independent Work and Side Projects: sensitive
16. Admin, Recovery, and System Operations: restricted

## Execution Model

Interactions may also be classified by execution style:

- `inline`
- `approval_gated`
- `delegated`

Examples:

- "Remember that I prefer aisle seats."  
  Memory and Knowledge + remember + inline

- "Draft a reply to this email."  
  Communication + plan + inline

- "Send this email to the vendor."  
  Communication + act + approval_gated

- "Research the best office chair options."  
  Decisions, Search, and Recommendations + act + delegated

- "Access finance notes for this session."  
  Finance and Assets + ask + approval_gated or grant-required

1. Home (dashboard)
2. Memory
3. Tasks
4. Calendar
5. Messages
6. Search

8. Health
9. My Home
10. Travel
11. People
12. Family
13. Hobbies
14. Community
15. Freelance

This is a rendering and product choice, not a different domain model.

## Why This Consolidation Exists

The unconsolidated conceptual surface is much larger than what is useful for:

- UI navigation
- product thinking
- permission systems
- delegated-task routing

The 16-domain model is intended to be the stable operational taxonomy. The full 99 interaction areas are documented in `docs/interaction/full_interaction_areas.md`.
