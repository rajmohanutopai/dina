Generated from CEO Plan Review (2026-03-20). Sequenced by dependency and impact.

| T01 | Fix malformed LLM JSON handling | P0 | S (human: ~2h / CC: ~15m) | — | Cloud LLMs (Gemini, Claude, OpenAI, OpenRouter) occasionally return HTTP 200 OK but with a garbled or incomplete JSON body — truncated mid-token, HTML error pages, or empty strings. Currently, the brain's Entity Vault path (`entity_vault.py → scrub_and_call()`) catches `LLMError` but a malformed 200 response may slip through as an unhandled `JSONDecodeError` or `KeyError`, crashing the PII rehydration step. This is dangerous because a crash mid-rehydration could leave partially-scrubbed data in an inconsistent state. The fix is to wrap JSON parsing in each of the 5 LLM adapter's `complete()` method (`llm_gemini.py`, `llm_claude.py`, `llm_openai.py`, `llm_openrouter.py`, `llm_llama.py`) with explicit try/except that catches `JSONDecodeError`, `KeyError`, and `TypeError`, converting them into a structured `LLMError("malformed_response", raw_body_preview=first_200_chars)`. The Entity Vault already handles `LLMError` correctly (clears the ephemeral vault, refuses the cloud call). Each adapter also needs a unit test with a garbled response fixture to prevent regression. ~10 lines per adapter. |

| T04 | Document CLIENT_TOKEN as known limitation | P0 | S (human: ~1h / CC: ~10m) | — | Dina's architecture documents (ARCHITECTURE.md, SECURITY.md, security-walkthrough.md) describe the target auth model as "Ed25519 everywhere — no JWTs, no shared secrets." However, the admin web UI still uses `CLIENT_TOKEN` — a 32-byte random bearer token generated during device pairing, used by the browser to authenticate session cookies. This is a Phase 1 compromise: the browser can't do Ed25519 natively, so Brain's admin app bridges with CLIENT_TOKEN. The problem is that the docs don't clearly call this out as a known gap. The fix is to add a "Known Phase 1 Limitations" section to SECURITY.md that explicitly states: (1) CLIENT_TOKEN exists for browser admin auth, (2) it's acceptable for single-user Home Node where Docker access = machine access, (3) Phase 2 replaces it with Ed25519 via a dedicated dina-admin backend, and (4) it is NOT acceptable for multi-tenant or remote-Brain deployments. |

| T13 | Write the Dina Protocol Specification | P3 | M (human: ~2w / CC: ~2d) | T06 | The Dina protocol is currently implemented in ~50,000 lines of Go, Python, and TypeScript code across 3 services. The wire formats, authentication flows, trust record schemas, and agent safety mechanisms are all working — but they exist only as code, not as a readable specification. This means: a Rust developer cannot build a Dina-compatible agent without reading Go source code; a security auditor must reverse-engineer the auth model from middleware implementations; the AT Protocol trust record schemas are defined in code but not documented as formal lexicons. The fix is to extract the protocol into 6 standalone specification documents in `docs/protocol/`: **(1) Identity** — How to create a Dina identity: generate a 32-byte master seed, convert to BIP-39 mnemonic, derive Ed25519 keys via SLIP-0010 at path `m/9999'/0'/0'`, create a `did:plc` via AT Protocol PDS or a `did:key` for standalone mode. DID Document format with `verificationMethod` and `serviceEndpoint`. Key rotation via signed PLC operations. **(2) Authentication** — The Ed25519 canonical signing format: `{METHOD}\n{PATH}\n{QUERY}\n{TIMESTAMP}\n{NONCE}\n{SHA256_HEX(BODY)}`. 5-minute timestamp window. 16-byte random nonce per request. Double-buffer nonce cache for replay protection. Per-service allowlists (brain, admin, connector). Headers: `X-DID`, `X-Timestamp`, `X-Nonce`, `X-Signature`. **(3) Messaging** — NaCl `crypto_box_seal` envelope format. Ed25519-to-X25519 key conversion for encryption. DIDComm-shaped plaintext: `{id, type, from, to, created_time, body}`. Dead drop semantics (store encrypted blob when vault locked, sweep on unlock). TTL handling. **(4) Trust** — All 19 `com.dina.trust.*` AT Protocol record types with full Zod/JSON Schema definitions. Trust score formula: `0.85 × (sentiment×0.40 + vouch×0.25 + reviewer_quality×0.20 + network×0.15) + 0.15 × 0.1`. Sybil defense mechanisms. Subject identity resolution (3-tier). **(5) Agent Safety** — Intent submission JSON format. 4-tier risk classification (SAFE/MODERATE/HIGH/BLOCKED) with the exact action sets. Session scoping. Approval flow. The "Deterministic Sandwich" pattern. **(6) PII** — 3-tier scrubbing pipeline (regex → NER → LLM). Entity Vault ephemeral pattern. Egress scanning rules. Each spec document should include: wire format examples, sequence diagrams, error codes, and security considerations. The protocol spec is what turns Dina from "a project" into "a standard." |

Useful for people who want work emails separate from personal. **Option 3: "Full privacy"** — Creates `personal` (default), `work` (standard), `health` (sensitive — every access logged + user notified), `financial` (locked — passphrase required, brain denied). Maximum compartmentalization. Each option auto-creates the personas with appropriate tiers, passphrases (prompted for sensitive/locked), and a confirmation showing what was created. The wizard can be re-run or personas can be manually managed via `dina-admin persona` commands. The principle: progressive disclosure. The complexity exists from day one (the architecture supports it), but the user sees only the complexity they're ready for. |
| T23 | Implement prompt injection Layer 1 | P7 | M (human: ~1w / CC: ~2d) | T01 | Layer 1 of the 7-layer prompt injection defense pipeline: input screening. Before any user-supplied text reaches the LLM, a lightweight classifier scans for common injection patterns. This is the first line of defense — cheap, fast, and catches the obvious attacks. Implementation: Create `brain/src/service/input_screener.py` with a `screen_input(text: str) -> ScreenResult` function. The screener checks for: (1) **Known injection patterns** — regex matches for "ignore previous instructions," "system prompt:", "you are now," "forget everything above," and similar well-documented injection phrases. (2) **Structural anomalies** — text that contains markdown code blocks with system-level instructions, XML-like tags that mimic system prompts, or unusual Unicode characters used to bypass regex (homoglyph attacks). (3) **Length anomalies** — input that's suspiciously long for the context (a simple question shouldn't be 10,000 characters). The screener returns one of: `CLEAN` (proceed normally), `SUSPICIOUS` (proceed with reduced tool access — the guardian limits which vault personas and MCP tools are available), or `BLOCKED` (reject the input, log the attempt, return a generic "I can't process that request" to the user). **Critical design principle: fails open.** If the screener itself fails (regex engine error, timeout), the system proceeds with a `SUSPICIOUS` flag rather than blocking. A broken screener should not prevent the user from using their own AI. The screener is called in `guardian.py → process_event()` before any LLM call. Audit log records every SUSPICIOUS and BLOCKED classification. |
| T24 | Implement prompt injection Layer 3 | P7 | M (human: ~1w / CC: ~2d) | T23 | Layer 3 of the 7-layer defense: per-stage tool isolation. The brain's guardian loop processes events through multiple stages (classification → vault query → LLM reasoning → nudge assembly → delivery). Currently, each stage has access to all tools. A prompt injection that compromises the LLM during the reasoning stage could potentially call tools meant for other stages (e.g., triggering vault writes during a read-only query, or calling MCP tools during nudge assembly). The fix is to implement per-stage tool allowlists. Each stage of the guardian loop declares which tools it's allowed to use: **Classification stage** — no tools (pure LLM text classification). **Vault query stage** — `search_vault`, `browse_vault`, `list_personas` only. **Reasoning stage** — `search_vault`, `browse_vault` (read-only), plus MCP tools from the allowlist (T25). **Nudge assembly stage** — no tools (operates on already-gathered context). **Delivery stage** — `notify` only. Implementation: Create a `ToolPolicy` class that wraps the LLM's function calling. Before each stage, set the active policy. If the LLM requests a tool not in the current policy, raise `ToolNotAllowedError`, log the attempt (this is a potential injection signal), and continue with the text the LLM has generated so far. Additionally, each stage gets a **fresh LLM context** — the conversation history from the previous stage is not carried forward. This prevents a multi-turn injection where the attacker builds up context across stages. The stage boundary is a context wipe — each stage sees only its own system prompt and the structured input from the previous stage's output. |
| T25 | Implement MCP tool allowlist | P7 | S (human: ~2d / CC: ~2h) | — | Layer 4 of the 7-layer defense: MCP (Model Context Protocol) tool allowlist. Brain delegates external work to child agents via MCP — OpenClaw for Gmail/Calendar, web search, etc. Currently, there's no restriction on which MCP tools the brain can call. If the LLM is prompt-injected during reasoning, it could potentially call any MCP tool, including ones that haven't been audited. The fix is simple and high-leverage: add a hardcoded allowlist `ALLOWED_MCP_TOOLS = {"fetch_emails", "fetch_calendar", "web_search", "fetch_contacts"}` in `brain/src/adapter/mcp_stdio.py`. Before any MCP `call_tool` invocation, check the tool name against the allowlist. If not allowed: log the attempt (tool name, calling context, LLM prompt that triggered it), raise `MCPToolBlockedError`, and return a generic "tool not available" response to the LLM. The allowlist is hardcoded, not configurable — this is intentional. Configurability means the LLM could potentially manipulate the config. A compiled-in list is auditable, testable, and immune to prompt injection. New MCP tools are added by code change + code review, not by configuration. This is ~20 lines of code that blocks an entire class of attacks (a compromised LLM calling `delete_emails`, `send_sms`, or any other unvetted MCP tool). |
| T26 | Complete D2D signature verification | P7 | M (human: ~1w / CC: ~2d) | — | The Dina-to-Dina (D2D) messaging system uses NaCl `crypto_box_seal` for encryption (working) and Ed25519 for message signing (stubs). The TODO.md notes that signature verification is not fully implemented. This means: encrypted messages are confidential (only the recipient can read them), but they are not authenticated — a node could forge a message claiming to be from any DID. The fix is to complete the signature verification in `core/internal/service/transport.go → ProcessInbound()`. The full flow should be: (1) Decrypt the NaCl sealed box using the recipient's X25519 private key (already working). (2) Parse the plaintext JSON to extract the `from` DID and `signature` field. (3) Resolve the sender's DID document to get their Ed25519 public key (via PLC directory or local contact cache). (4) Verify the Ed25519 signature over the message body (excluding the signature field itself). (5) Check for replay: `senderDID|msgID` must not exist in the replay cache (SEC-HIGH-08, 24-hour window). (6) If signature verification fails: reject the message, log the attempt with the claimed sender DID, and optionally add the source IP to the ingress blocklist (Valve 3 sweeper feedback). (7) If the sender's DID cannot be resolved: quarantine the message (store with `Quarantined=true` flag for later review when the DID might become resolvable). This is a critical security fix — without signature verification, the D2D messaging system's trust model is incomplete. Encryption without authentication is like a locked box with no return address — you know no one read it in transit, but you don't know who sent it. |
| T27 | Supply chain security | P7 | M (human: ~1w / CC: ~2d) | — | Dina is a sovereign system that holds your most sensitive data. If an attacker compromises the container images (supply chain attack), they could exfiltrate vault keys, inject malicious code into the PII scrubber, or modify the gatekeeper rules. The architecture docs specify three defenses, none yet implemented: **(1) Digest pinning** — Every image reference in `docker-compose.yml` must use a full SHA-256 digest instead of tags. Change `image: ghcr.io/dina/core:latest` to `image: ghcr.io/dina/core@sha256:a1b2c3...`. This ensures that `docker compose pull` always fetches the exact image the user verified, not whatever `:latest` points to today. **(2) Cosign image signing** — Set up Sigstore Cosign keyless signing in GitHub Actions CI. Every image built by the release workflow is signed using the GitHub OIDC identity (no manual key management). Users verify before upgrade: `cosign verify ghcr.io/dina/core@sha256:... --certificate-identity=https://github.com/rajmohanutopai/dina/...`. This proves the image was built by the official CI, not by an attacker. **(3) SBOM generation** — Use `syft` to generate an SPDX Software Bill of Materials for each container image. Attach the SBOM to the image via Cosign. Users can scan for known vulnerabilities: `grype ghcr.io/dina/core@sha256:...`. **(4) No auto-updates** — Dina never pulls new images without user knowledge. The user initiates upgrades. `dina-admin upgrade` checks for new versions, shows the Cosign verification status, and only pulls after confirmation. Add verification instructions to `QUICKSTART.md`. The principle: sovereignty means the user controls every binary that runs on their Home Node. Trust is verified by math (signatures), not by faith ("`latest` is probably fine"). |
